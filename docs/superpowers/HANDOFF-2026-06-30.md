## The KEY strategic finding (most important takeaway)

The experts asked to push rules to 90% precision and "activate" (score) them. The honest verdict, proven by rigorous measurement:

- **Detection rules** (value/pattern: `color`, `no-arbitrary-tailwind`, `interactive-role-name`, …) → **cannot reach 90% deterministically.** Two careful proofs both passed synthetic adversarial validation (J=1, Wilson LB ~0.90) then FAILED real-corpus confirmation with a fair judge (22% and 6% real precision) because the false-positive space is SEMANTIC and open-ended — real code always has FP classes fixtures can't enumerate (the "color wall", now shown to generalize). The corpus-confirmation guard works — it catches these.
- **Structural rules** (decidable facts: does a story document props? does a manifest list examples?) → **CAN reach 90%** (no semantic gap). `stories/props-documented` was tightened (only fire when the component HAS props) and hardened to a clean synthetic proof (precision/recall Wilson LB **0.9011**, N=46, J=1).

The path for detection rules is therefore NOT deterministic promotion — it's the **LLM precision filter (#115, already in the product)** at audit time (Solution 2).

`props-documented` is the closest to promotable (clean, deterministic) but its **corpus confirmation hit three INFRASTRUCTURE walls** (not rule defects): consumer-app repos keep DS components in node_modules; the rule skipped dsSelfMode; the inventory matches PascalCase filenames but modern DS repos use lowercase `button/index.tsx`. Each was a measurement-infra limit, not a rule fault.

## Key facts / gotchas for whoever resumes

- **Promotion = score change = always post-merge** (Phase B). Pre-merge you can only MEASURE who's promotable.
- **Synthetic adversarial J=1 is OPTIMISTIC for detection rules** — only trust it for genuinely deterministic (no-semantic-gap) rules. The corpus-confirmation guard exists to catch the difference; honor it.
- **The catalogue-coherence keystone test** forces every catalogue number to equal `deriveMeasurement(adapter matrix)` — no hand-pasted constants. To keep a rule honestly null, its adapter must have NO `falseFriends`.
- **Adding `falseFriends` to an adapter forces measured (non-null) catalogue numbers** — that's "measured but off-score", still experimental until Phase B flips `contributesToScore`.
- **The LLM judge runs via the agent-CLI connector (no API key)** but batches must be small; it needs a PER-RULE rubric (fixed — sources from rule `meta`).
