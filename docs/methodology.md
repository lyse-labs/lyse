# Lyse Methodology

> How Lyse measures AI-readiness in design systems — and why.

---

## The Two-Faces Model

Every design-system AI guideline in circulation today conflates two fundamentally different questions. Lyse separates them, because a design system can score well on one and fail catastrophically on the other.

**Face A — AI-Consumable.** Can an AI agent (Claude, Cursor, Copilot) understand and use this design system *without hallucinating*? This is about machine-readable tokens, `AGENTS.md` files, `llms.txt`, component manifests, typed contracts — the plumbing that lets agents consume the system faithfully.

**Face B — AI-Governance.** Does the design system encode *how to build AI features responsibly*? Reserved marking tokens, layered explainability, human-control affordances, failure design, feedback controls, disclaimers — the rules that converge across IBM Carbon, AWS Cloudscape, GitLab, Microsoft Fluent, and the canonical HAX / PAIR frameworks.

> **Why separate?** A system can have perfect DTCG tokens (Face A) while having zero AI-marking guidelines (Face B). Conversely, a system can have a beautiful `AILabel` component (Face B) while its tokens are hardcoded hex values agents will hallucinate (Face A). Both faces matter. One number hides problems.

## Face A: AI-Consumable

The signals. Each maps to sources in Appendix B.

### A.1 Machine-readable tokens

DTCG format (`$value`, `$type`, `$description`, `$schema`). `$references` resolve across files. Agents consume the schema directly — hallucination risk drops when the contract is machine-readable.

**Sources:** Design Tokens Community Group spec

### A.2 Agent instruction files

- `AGENTS.md` — runnable commands, exit codes, toolchain references
- `llms.txt` / `llms-full.txt` — H1 + blockquote + section structure per llmstxt.org
- Component manifests (`components.json`)

**Sources:** llmstxt.org, Anthropic MCP, Spotify "AI-ready DS"

### A.3 Component contracts

Typed props (no `any`), exported union variants, shipped `.d.ts` files. Strictness matters: `string` props let agents guess; union types constrain them.

**Sources:** shadcn registry, Storybook manifests (react-docgen-typescript)

### A.4 Design-to-code bridge

- Figma Code Connect coverage (`*.figma.tsx`, `figma.config.json`)
- MCP server presence (`.mcp.json`)
- DS index export surface
- Storybook story coverage

**Sources:** Figma Code Connect, Anthropic MCP, Storybook

## Face B: AI-Governance

*The differentiator — no one automates this at scale.*

### B.1 Marking *(always label AI outputs)*

Reserved tokens (`ai-*`, `--p-color-*-magic*`, `*-color-*-ai-*`). Dedicated marker components (`AILabel`). Composite rule: **AI token used without marker = fail** — this is IBM Carbon's mandatory rule and the single strongest convergent signal across top design systems.

**Critical:** The sparkle/magic icon is strictly reserved for generative AI. Using it as decoration is an anti-pattern (SAP, AWS Cloudscape).

**Sources:** IBM Carbon, AWS Cloudscape, Shopify Polaris, Workday Canvas, GitLab Pajamas

### B.2 Layered explainability *(What → Why → How)*

| Layer | What the user sees | Who it's for |
|-------|-------------------|-------------|
| Surface | "AI-generated" badge + brief summary | End user |
| Detail | Reasoning, sources, confidence | Developer / auditor |
| Deep | Model inputs, process, limitations | Regulator, researcher |

No layer dumps model internals. Depth scales with decision stakes.

**Sources:** IBM Carbon, AWS Cloudscape, SAP Fiori

### B.3 Human control *(the human decides, the human can undo)*

Affordances present: stop / edit / regenerate / undo / confirm / delete. Destructive actions require explicit authorization. Risk-tiered: simple confirmation for low-risk, explicit authorization for destructive or cascading changes, always offer reversal.

**Sources:** GitLab Pajamas, Microsoft Fluent 2, Apple HIG, AWS Cloudscape

### B.4 Failure & failure-recovery

- AI-specific error state (not a generic error)
- Named loading states: "Generating…" with paired text
- Non-AI fallback path (GitLab signature)
- Access to human support

**Critical:** A dead-end error (no retry, no degrade, no human) is an automatic fail.

**Sources:** PatternFly AI, Cloudscape gen-AI, GitLab Pajamas, HAX G14

### B.5 Feedback controls

Thumbs, rating, or categorized feedback on AI output. Both explicit (user-reported) and implicit (telemetry) signals. Categories must be meaningful: harmful, inaccurate, biased — not "other".

**Sources:** Microsoft Fluent 2, HAX G15, PAIR, PatternFly

### B.6 Disclaimers & attribution

Near AI output: "AI-generated / may be incorrect". GitLab exact string: "Content generated by AI should be seen as a starting point and verified before use." Never hide AI involvement.

**Sources:** GitLab Pajamas, Microsoft Fluent 2, Emplifi Soul

## The Qualitative Layer

Five dimensions the engine cannot statically detect. These are scored by LLM judgment, validated by the Bench:

1. **Human control *actually enforced*** — controls present ≠ wired to the AI output
2. **Voice/tone & anti-anthropomorphism** — agents are not team members; no emotions, no social bonds
3. **Explanation quality** — a popover exists ≠ its content is useful
4. **Risk classification** — is this AI feature low-risk chat or high-stakes autonomous action?
5. **Is AI even needed?** — the value gate (ServiceNow 10-question model)

## Maturity Model

Lyse maps its score onto [Kavcic's 5-level maturity model](https://learn.thedesignsystem.guide/p/design-systems-that-document-ai):

| Level | What the DS documents | Lyse score |
|-------|----------------------|------------|
| **0** Nothing | No AI guidance visible | Below detection threshold |
| **1** Decoration | Color tokens / badges, no rules | 0–20 % |
| **2** Components | Reusable `AILabel`, chat panel | 20–40 % |
| **3** Behaviors | Guidelines for AI actions (suggest, respond, handoff) | 40–60 % |
| **4** Governance | Enforced rubrics, gate, letter grades | 60–80 % |
| **5** Infrastructure | Cross-platform AI component kits | 80–100 % |

The critical jump is **Level 2 → 3**: from styling AI to governing how it *behaves*.

---

## How we keep it honest: the Bench

The Bench continuously verifies static-rule scores against LLM-expert grades, per dimension, using Cohen's κ. A static rule that doesn't track the expert is flagged — not surfaced as truth. Sub-axes only promote from `experimental` to `stable` when the Wilson 95 % lower bound ≥ 0.90 on the calibration corpus.

This means Lyse can add new signals **without eroding trust**: every rule passes define → TDD → calibrate → validate → promote.

> The Bench is itself the first implementation of **design-system observability** — one of Kavcic's twelve undocumented areas.
