# MCP server

Wire Lyse into Cursor, Claude Code, Codex, or any [Model Context Protocol](https://modelcontextprotocol.io/)-compatible IDE.

## Why

Without Lyse, your AI agent writes UI code without knowing your design system. It hallucinates components, hardcodes colors, ignores accessibility — silently, every day.

With Lyse's MCP server, your agent can call two tools:

- Audit a file it's about to save (`audit_file`).
- Request a unified diff to fix a finding (`suggest_fix`).

Result: fewer DS violations in AI-generated PRs, fewer review cycles, less drift.

## Setup

### Automated setup (recommended)

Run `lyse mcp setup` to auto-detect your IDE and write the correct config block:

```bash
lyse mcp setup
```

The command detects Cursor, Claude Code, or VS Code and appends the `lyse` server entry to the correct file. Pass `--ide=cursor` / `--ide=claude-code` to override detection.

> **Dev-mode auto-detection.** When you run `lyse mcp setup` from a local checkout (e.g. `node packages/core/dist/cli.js mcp setup`), Lyse detects that it is not running from an npm install and writes an absolute-path entry instead of the `npx -y lyse mcp` default. You can also force this behavior with `--dev`. This ensures contributors get a working MCP server entry without manually editing `.mcp.json`.
>
> Use `--dev` as the escape hatch if your Lyse checkout happens to live under a `node_modules/` ancestor (e.g. a fork linked into another project via `pnpm link`). The auto-detection will read that ancestor as "installed via npm"; `--dev` forces the absolute-path entry regardless.

## Running the server

The MCP server is started by your IDE via the entry `lyse mcp setup` writes — you don't normally invoke it by hand. Two equivalent ways to start the stdio server:

```bash
lyse mcp           # default — no subcommand
lyse mcp serve     # explicit — same behavior, useful for clarity in scripts
```

Both speak JSON-RPC 2.0 over stdin/stdout per the MCP spec. Setup entries generated by `lyse mcp setup` use the `mcp` form for backwards compatibility; the `serve` subcommand exists so that `lyse mcp --help` exposes the stdio mode explicitly.

### Manual setup — Cursor

Add to `.cursor/mcp.json` at the project root:

```json
{
  "mcpServers": {
    "lyse": {
      "command": "npx",
      "args": ["-y", "lyse", "mcp"]
    }
  }
}
```

Restart Cursor. The Lyse tools become available in the agent's tool palette.

### Manual setup — Claude Code

Add to `.mcp.json` at the project root (same format as Cursor):

```json
{
  "mcpServers": {
    "lyse": {
      "command": "npx",
      "args": ["-y", "lyse", "mcp"]
    }
  }
}
```

Restart your Claude Code session. Run `/mcp` to confirm `lyse` is connected.

### Codex / other MCP clients

Use the same command (`npx -y lyse mcp`) and stdio transport. Refer to your client's MCP configuration docs for the file path and format.

### Verify the connection

In your IDE chat, ask:

> "Use the lyse tool to audit the current file."

The agent should call `audit_file` and return a Health Score for that file.

If nothing happens:

- Confirm `npx -y lyse mcp` works on the command line.
- Check the IDE's MCP logs (Cursor: View → Output → MCP).
- Run with `LYSE_DEBUG=1` for verbose logging.

## Tools exposed

### `audit_file(path, content?)`

Audit a single file. The optional `content` lets the agent pass an unsaved buffer.

**Parameters:**

| Param | Type | Required | Description |
|---|---|---|---|
| `path` | string | yes | Absolute or relative path to the file. |
| `content` | string | no | If provided, audit this content instead of reading the file from disk. |

**Returns:**

```json
{
  "score": 73,
  "findings": [
    {
      "rule_id": "tokens/no-hardcoded-color",
      "rule_version": "v1",
      "line": 17,
      "column": 24,
      "snippet": "background: \"#3B82F6\"",
      "severity": "warning",
      "help_uri": "https://github.com/lyse-labs/lyse/blob/main/docs/rules/tokens-no-hardcoded-color.md"
    }
  ]
}
```

**Use case:** the agent has just generated a new component. Before saving, it calls `audit_file` with the buffer content, sees 2 findings, fixes them, and only then writes the file.

**Rule coverage:** `audit_file` runs the rules that produce meaningful results from a single file in isolation — `tokens/no-hardcoded-color`, `tokens/no-hardcoded-spacing`, `components/no-native-shadows`, `a11y/essentials`, `naming/component-pascalcase`, and `naming/hook-prefix`. Repo-wide rules (governance, AI-surface, story coverage) need the full project graph and only run under `lyse audit`; they will be exposed to agents via MCP resources in a later release.

### `suggest_fix(path, rule_id, line)`

Request an auto-fix for a specific finding.

**Parameters:**

| Param | Type | Required | Description |
|---|---|---|---|
| `path` | string | yes | File to fix. |
| `rule_id` | string | yes | The rule ID from a previous `audit_file` finding. |
| `line` | integer | yes | The line number from the finding. |

**Returns:**

```json
{
  "diff": "--- a/src/Button.tsx\n+++ b/src/Button.tsx\n@@ -17,1 +17,1 @@\n-  background: \"#3B82F6\",\n+  background: tokens.color.brandPrimary,",
  "imports_added": [
    "import { tokens } from \"@your-org/ui/tokens\";"
  ],
  "confidence": "high"
}
```

**Confidence levels:**

- `high` — codemod is deterministic; the diff is safe to apply.
- `medium` — codemod uses heuristics; review before applying.
- `low` — codemod was unable to produce a fully-resolved fix; the diff is a suggestion.

**Auto-fixable rules:**

- `tokens/no-hardcoded-color`
- `tokens/no-hardcoded-spacing`
- `components/no-native-shadows`
- `naming/component-pascalcase`
- `naming/hook-prefix`

For non-auto-fixable rules, `suggest_fix` returns a structured error explaining why.

## How agents should use these tools

### Default pattern

For most AI code generation tasks involving UI:

1. **After writing:** call `audit_file` with the buffer content.
2. **If findings exist:** call `suggest_fix` for each fixable finding and apply the diff.
3. **For non-fixable findings:** ask the user how to proceed.

### `AGENTS.md` integration

Commit an `AGENTS.md` file to your repo:

```bash
npx @lyse-labs/lyse agents > AGENTS.md
```

This summarizes:

- Available rules.
- The components module (`@your-org/ui`).
- The intent map.
- Which rules are auto-fixable.

Cursor / Claude Code / Codex read `AGENTS.md` on session start. Together with the MCP server, this gives the agent a complete picture: what the rules are, which to call, what's auto-fixable.

## Latency

Local stdio MCP calls are fast:

| Call | Typical latency |
|---|---|
| `audit_file` (small file, < 200 lines) | 50–150 ms |
| `audit_file` (large file, 1000+ lines) | 200–500 ms |
| `suggest_fix` | 30–100 ms |

The first call after agent startup is slower (200–800 ms) due to module loading.

## Privacy

The MCP server runs **fully locally**. No data leaves your machine.

- Tool calls happen over stdio between the IDE and the local `lyse` process.
- File contents passed to `audit_file` are parsed in-memory and discarded.
- No telemetry is sent for MCP usage (telemetry, if enabled, only covers the CLI `lyse audit` flow).

## Limitations

- **React (TSX/JSX) only.** Vue / Svelte / Solid / Angular are not yet supported.
- **No streaming output.** Each tool call returns a single response. Streaming may be added in a future MCP spec.
- **No multi-file analysis** in a single call. Each `audit_file` is scoped to one file.
- **No persistent state between calls.** The MCP server is stateless. Configurations are re-read on every call (cached for ~1 second).

## Troubleshooting

**Q: My agent doesn't call the Lyse tools.**
The agent needs to know they exist. Commit `AGENTS.md` (generated by `lyse agents`). Also: many agents need to be explicitly prompted ("Use the lyse tool to audit this file").

**Q: `audit_file` returns a different score than `lyse audit`.**
The CLI scores the whole repo; the MCP tool scores one file. That's expected.

**Q: The MCP server hangs.**
Run `npx -y lyse mcp` in a terminal manually. If it hangs there too, it's a Lyse bug — please file an issue with `LYSE_DEBUG=1` output. If it works in the terminal but not in the IDE, the issue is in the IDE's MCP integration.

**Q: Can I run the MCP server remotely?**
Not today. The server uses stdio transport.

**Q: Does this work offline?**
Yes — after the initial `npx -y lyse mcp` install (which fetches the package), the server runs entirely offline.
