Fixed the TypeScript parser selecting the wrong syntax for `.jsx` files and for Vue/Svelte single-file-component `<script>` blocks. `parseTs` derived swc's options from the file extension: `.jsx` was parsed with the legacy `jsx` flag (which swc ignores under the `typescript` syntax — `tsx` is the real lever), so any `.jsx` file containing JSX threw and was silently dropped; and SFC scripts arrive with their `.vue`/`.svelte` path, which fell through to the `ecmascript` syntax, so any typed `<script lang="ts">` threw. Both silently produced an empty AST (no imports, no CSS-in-JS) and were then miscounted as "malformed source" parse errors — zeroing DS-import detection (`components/shadow-native`), CSS-in-JS token scanning, and component/token coverage for the most common non-React design-system stacks (Vue/Nuxt, SvelteKit) auditing their own source. The parser now always uses the `typescript` syntax (JavaScript is a syntactic subset) and toggles `tsx` on for `.tsx`/`.jsx`/`.vue`/`.svelte`, keeping it off for plain `.ts` so angle-bracket type assertions (`<T>x`) still parse. Golden corpus byte-identical: none of the four tokenize via SFC `<script>` blocks, and although the fix does flip `.jsx` parse outcomes on the tailwind-dashboard template (43 files that previously threw), that repo declares no `componentsModule` and no CSS-in-JS, so its scored output is unchanged. Real Vue/Svelte/JSX design systems auditing their own source will now report accurate findings where they previously silently reported none.
